# 编码规范

> 这是模板骨架，请替换成你自己项目的真实规范。不要把这里当成固定真相。

## 1. 修改原则

- 只改与当前需求直接相关的代码。
- 优先跟随仓库现有风格，而不是强行引入新偏好。
- 新增抽象前先确认是否真的复用。
- 如果发现无关问题，记录下来，不顺手大改。

## 2. 命名建议

| 类型 | 建议 |
|------|------|
| 组件 | PascalCase |
| Hook | `useXxx` |
| 工具函数 | 动词开头，表达单一职责 |
| 常量 | `UPPER_SNAKE_CASE` |
| 类型/接口 | 与项目现有习惯保持一致 |

## 3. 文件组织建议

- 按责任拆文件，不按“我顺手”拆文件。
- 变更会一起发生的代码尽量靠近。
- 如果一个文件已经很大，优先做局部边界梳理，而不是全盘重构。

## 4. 测试与验证

- 改动前先想清楚成功标准。
- 能补自动验证就补自动验证。
- 如果无法自动化验证，至少写清楚人工验证步骤。

## 5. 文档要求

- 重要决策写入 `design.md`。
- 可执行任务写入 `tasks.md`。
- 长期有效的工程知识回写到 `vertical-knowledge/`。
