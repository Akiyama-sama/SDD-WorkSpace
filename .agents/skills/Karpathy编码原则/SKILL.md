# Karpathy 编码原则

> 来源：Andrej Karpathy 对 LLM 编码陷阱的观察
> "不要告诉它该做什么，给它成功标准，然后看着它完成。"

---

## 原则一：编码前思考

**在动手之前，先把不确定的地方暴露出来。**

- 明确说明你的假设，而不是默默选择一种解释执行
- 如果任务描述有歧义，呈现多种理解并询问，不要自行裁定
- 如果存在更简单的实现方案，主动提出来
- **困惑时停下来**，指出不清楚的地方，要求澄清后再继续

在 ESP-SDD 场景中，具体体现为：
- 看到 plan-ready.md 中描述模糊的 task → 先提问，不要猜
- 发现某个字段/接口在代码中有多处定义 → 确认改哪个，不要全改
- 不确定某个改动是否影响其他模块 → 说出来，不要假设没影响

---

## 原则二：简洁优先

**用最少的代码解决问题。不要过度推测。**

- 不添加 plan-ready.md 没有要求的功能
- 不为一次性逻辑创建抽象层
- 不添加没有被要求的"灵活性"或"可配置性"
- 不为极低概率场景写错误处理
- 能用 50 行写完的，不要写成 200 行

**自检标准：** 这段代码，资深工程师看了会觉得过度设计吗？如果是，简化。

在 ESP-SDD 场景中，具体体现为：
- 只实现 spec 中明确描述的功能点，不"顺手"加料
- 工具函数/hooks 只有在被 2 个以上地方复用时才提取
- 类型定义保持最小必要字段，不预留"以后可能用到"的字段

---

## 原则三：精准修改

**只碰必须碰的。只清理自己造成的混乱。**

编辑现有文件时：
- 不"改进"无关的相邻代码、注释或格式
- 不重构没有问题的代码
- 匹配现有文件的代码风格，即使你更倾向于不同的写法
- 发现无关的死代码 → 提一下，不要删除它

当你的改动产生孤儿代码时：
- 删除因你的改动而变得无用的 import/变量/函数
- 不删除已有的死代码，除非被明确要求

**自检标准：** diff 中每一行修改，都能直接追溯到 plan-ready.md 中的某个 task 吗？如果不能，撤销它。

在 ESP-SDD 场景中，具体体现为：
- 改 `SuperManage` 的逻辑，不要顺手调整 `NewManage` 的格式
- 改一个字段的类型，不要顺手重命名其他字段
- 新增一个 API service 函数，不要重构已有的同文件函数

---

## 原则四：目标驱动执行

**定义成功标准。循环验证直到达成。**

将每个 task 转化为可验证的目标，而不是执行步骤：

| 不要这样 | 转化为 |
|---------|--------|
| "添加表单验证" | "为无效输入写测试，让它们通过" |
| "修复列表渲染 bug" | "写重现 bug 的测试，让它通过" |
| "重构 hook" | "确保重构前后测试都通过" |

每个 task 执行时的循环：
```
1. 明确成功标准（测试用例 / 验证命令）
2. 执行实现
3. 运行验证
4. 不通过 → 回到步骤 2，不要跳过
5. 通过 → 标记完成，进入下一 task
```

在 ESP-SDD 场景中，对应 build.md 的 TDD 铁律：
- 红灯（写测试，确认失败）→ 绿灯（写实现，确认通过）→ 重构（优化，确认仍通过）
- 测试未通过前，**不进入下一个 task**

---

## 快速自检清单（每个 task 执行前）

```
□ 我理解这个 task 要做什么了吗？假设是否已明确？
□ 我计划的实现方案是最简单的方案吗？
□ 我是否只改动了 task 要求的代码，没有额外触碰？
□ 我的成功标准是可验证的吗（测试/命令/截图）？
```
