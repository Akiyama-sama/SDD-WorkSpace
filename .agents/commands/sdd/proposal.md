# Proposal: 轻量需求捕获

## 目标

用最少的提问，把用户脑子里的需求变成可执行的变更描述。不做深度设计，不生成代码。

## 流程

### 0. 模式检测（入口分流）

执行任何操作前，先检测是否存在活跃变更：

```bash
ls openspec/changes/ 2>/dev/null | grep -v archive
```

- 无活跃变更 → `mode=create`，进入步骤 1
- 有活跃变更 → 进入模式确认门控

读取活跃变更目录下的 `proposal.md`，用一段话概括当前需求摘要，然后展示门控：

```text
检测到进行中的变更：<变更名>

当前需求：
  <2-3 行摘要>

你想做什么？
  A. 在此基础上补充/调整需求（amend 现有变更）
  B. 归档当前变更，新建一个变更
  C. 取消
```

- 用户选 A → `mode=amend`，进入步骤 1A
- 用户选 B → 提示用户先执行 `/sdd close` 归档当前变更，本次流程终止
- 用户选 C → 流程终止

> 严禁直接覆盖现有 `proposal.md`，所有修改必须经过 amend 分支的确认面板。

### 1A. 增量分支（mode=amend）

用一句自由描述就够了，不要强制结构化。

展示提问框：

```text
请描述要补充/调整的需求（一句话即可）：
  ▌
```

用户输入自由文本后，产出变更预览：

```text
我理解你想做以下调整：
  ① <变更点 1>      [新增功能点]
  ② <变更点 2>      [修改现有描述]
  ③ <变更点 3>      [删除原有功能点]

将合并到 proposal.md，并自动标记后续规格失效（需重新跑 /sdd spec 同步方案）。

是否确认？
  A. 确认，写入
  B. 我想修正描述
  C. 取消
```

- 用户选 A：
  1. 将变更点合并写入 `openspec/changes/<变更名>/proposal.md`
  2. 删除 `openspec/changes/<变更名>/plan-ready.md`（如存在），强制下次 spec 重生成
  3. 在变更目录下追加/更新 `.sdd-state.json`：`{"proposalLastEditedBy":"proposal:amend","planReadyStale":true}`
  4. 跳到步骤 5
- 用户选 B → 回到自由描述输入框
- 用户选 C → 流程终止

> amend 分支不走步骤 1/2/3/4，直接跳到步骤 5。

### 1. 识别输入类型

首先判断用户输入属于哪种类型，选择对应的处理路径：

#### 类型 A：自然语言描述

特征：用户用口语或简短文字描述需求，例如“我想加一个导出功能”。

处理：分析已包含的信息维度，只针对缺失部分提问：

- 缺失 0-1 个维度 → 直接进入步骤 3 确认
- 缺失 2-3 个维度 → 一次性追问缺失部分
- 缺失 4-5 个维度 → 提完整的 3-5 个问题

需要收集的五个维度：

| 维度 | 说明 |
|------|------|
| 做什么 | 想实现什么功能/变更？ |
| 为什么 | 解决什么问题？给谁用的？ |
| 成功标准 | 怎样算做完了？验收条件是什么？ |
| 边界 | 什么不在范围内？ |
| 现有约束 | 技术栈、兼容性、时间上的限制？ |

#### 类型 B：任务 / 工单 ID

特征：用户输入看起来是一个任务号，或明确说“这是个工单/任务”。

处理：
1. 若存在可用的任务系统连接器，读取任务详情
2. 从标题、描述、子任务中自动提取五个维度
3. 若无连接器，要求用户贴出关键摘要
4. 将提取结果作为草稿进入步骤 3 确认

#### 类型 C：需求文档

特征：用户粘贴了大段结构化文档，或提供了文档链接。

处理：
1. 如果有可用文档读取能力，则先读取内容
2. 否则要求用户粘贴关键章节
3. 从文档中自动提取五个维度，忽略技术实现细节
4. 将提取结果作为草稿进入步骤 3 确认

### 1.5 知识库查询（仅 mode=create）

如果当前仓库存在 `vertical-knowledge/`，在产出结构化需求前，先查询相关规则和模块知识，避免对术语、已有能力和边界的理解偏差。

输出应类似：

```text
知识库查询结果（proposal 阶段）：

  [相关模块]   <模块/功能> → <一句话摘要>
  [相似功能]   <已有功能> → <与本需求的差异/重合点>
  [关键术语]   <业务术语> → <对应业务概念>

  结论：知识库覆盖了以下背景：
    - <要点列表>
  待用户澄清的疑点：
    - <要点列表，无则填“无”>
```

查询失败或无结果时，不阻塞流程，直接继续。

### 2. 判断是否需要拆分

如果需求跨越多个独立子系统或功能域，先建议拆分，再分别走 proposal 流程。

拆分判断标准：
- 涉及 2 个以上相互独立的模块
- 不同子需求的验收方不同
- 任一子需求单独实现就能交付价值

### 3. 确认需求

整理成结构化需求描述，与用户确认：

```text
我理解的需求是：[一句话概括]

- What（做什么）：[描述]
- Why（为什么）：[描述]
- Done When（成功标准）：[描述]
- Out of Scope（边界）：[描述]
- 约束：[描述，无则填“无”]

这样理解对吗？
```

### 4. 创建 OpenSpec 变更

用户确认后，先检查目标目录是否已存在：

```bash
ls openspec/changes/<变更名>/
```

- 如果已存在，提示用户确认是否覆盖
- 如果不存在，优先调用：

```bash
openspec new <变更名>
```

若 CLI 不可用，则手动创建：

```bash
mkdir -p openspec/changes/<变更名>/specs
```

将步骤 3 确认的结构化需求描述写入 `openspec/changes/<变更名>/proposal.md`。

**变更命名规范**：推荐格式为：
- `<日期>-<短名称>`
- 若有工单号：`<日期>-<工单号>-<短名称>`

示例：
- `2026-06-21-add-export-center`
- `2026-06-21-T123456-add-export-center`

### 5. 提示下一步

**mode=create 时：**
> 「需求已记录。接下来可以用 `/sdd spec` 生成完整规格，或继续补充细节。」

**mode=amend 时：**
> 「需求变更已合并到现有 proposal。由于翻译层已失效，请用 `/sdd spec` 重新对齐技术方案。」

> STOP：proposal 流程到此结束。不得继续自动触发 `spec`、`brainstorming` 或任何代码生成阶段。

## 注意

- 不要做技术设计，那是 spec 和 brainstorming 的事
- 不要写代码
- 问题要具体，不要泛泛而谈
- 优先利用用户已提供的信息，减少不必要追问
- mode=create 时，优先利用知识库，但不得用知识库覆盖用户原意
- mode=amend 时，禁止重新跑完整五维度提问
