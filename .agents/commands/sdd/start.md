# Start: SDD 全流程编排

## 使用方式

```text
/sdd start [需求描述 或需求文档链接（可选）]
```

## 启动

宣告进入 SDD 流程：

```text
已启动 SDD 规范驱动开发流程。

正在检查环境与项目状态...
```

执行以下状态检测，决定从哪里开始：

### 状态检测（断点恢复）

```bash
# 1. 环境就绪？
cat .agents/sdd-state.json 2>/dev/null

# 2. 有活跃变更？
ls openspec/changes/ 2>/dev/null | grep -v archive

# 3. 有 plan-ready.md？
ls openspec/changes/*/plan-ready.md 2>/dev/null

# 4. 有轻量 spec.md？
ls openspec/changes/*/spec.md 2>/dev/null

# 5. 有执行计划文件？
ls openspec/changes/*/impl-plan.md 2>/dev/null
```

根据检测结果决定入口：

| 检测结果 | 行为 |
|---------|------|
| `sdd-state.json` 不存在或 `setupDone=false` | 从 setup 开始 |
| 环境就绪，无活跃变更 | 从阶段一（需求收敛）开始 |
| 有活跃变更，无规格 | 询问断点恢复，从阶段二（规格生成）继续 |
| 有 `plan-ready.md` 或 `spec.md`，且存在未完成计划 | 询问断点恢复，从阶段三（执行实现）继续 |
| 计划文件全部 checkbox 已勾选 | 询问断点恢复，从阶段四（验证归档）继续 |

### 断点恢复·分支一致性检查

恢复到阶段三或阶段四时，额外执行分支一致性检查：

```bash
# shellcheck source=/dev/null
source .agents/tools/sdd-config.sh
BASE_BRANCH="$(sdd_config_get_base_branch)"
CURRENT_BRANCH="$(git branch --show-current)"
printf '当前模板配置的受保护主分支：%s\n' "$BASE_BRANCH"
printf '当前分支：%s\n' "$CURRENT_BRANCH"
```

检测当前分支是否为受保护主分支（当前模板配置：`main`）：

- 若当前在受保护主分支 → 提示用户切换到开发分支后继续
- 若当前在其他分支 → 提示用户确认这是本次需求的开发分支

断点恢复提示格式：

```text
检测到进行中的变更：<变更名>
当前状态：<阶段描述>

是否继续上次进度？
  A. 继续（从上次中断处开始）
  B. 重新开始新变更
```

### 步骤 0：需求分流（硬关卡）

> 进入阶段一之前必经。已存在 triage 结果时可跳过，否则强制执行。

**0.1 收集需求信息**

- 用户已带需求描述或文档链接 → 直接读取
- 用户未带 → 提示：「简单描述一下你要做什么」

**0.2 自动信号扫描**

读取需求文本，命中以下关键词时记录信号：

| 关键词 | 信号倾向 |
|--------|----------|
| 文案 / 样式 / 排版 / 小交互 | 简单 |
| 权限 / 灰度 / 发布 / 配置 / 监控 | 中等 |
| 新增字段 / 改 schema / 加接口 / 状态机 | 复杂 |
| 多仓库 / 跨模块 / 跨团队 / 第三方集成 | 复杂 |
| 认证 / 支付 / 账务 / 数据迁移 / 不可逆 | 强制复杂 |

**0.3 PD 估算**

估算开发人天（PD）。用户已明确给出则直接用，否则给出 AI 估算并让用户确认。

**0.4 5 维复杂度评分**（每项 0/1/2 分）

| 维度 | 0 分（低） | 1 分（中） | 2 分（高） |
|---|---|---|---|
| ① 影响面 | 单文件 / 单组件 | 单模块内多文件 | 跨模块 / 跨仓库 |
| ② 数据契约变更 | 无 | 改前端接口/枚举 | 改后端 schema / 状态机 |
| ③ 依赖耦合 | 无外部依赖 | 调用现有接口 | 新增集成 / 改调用链 / 跨团队联调 |
| ④ 可逆性 | 改错好回滚（纯 UI） | 需要灰度 / 配置 | 数据 / 账务 / 不可逆副作用 |
| ⑤ 不确定性 | 方案明确 | 有 1~2 个待定方案 | 需调研 / 历史代码不熟 / 文档模糊 |

总分：0~2 → 低 / 3~5 → 中 / 6~10 → 高

**强制升档规则**：维度 ② 或 ④ 任一打到 2 分 → 强制 `full`，禁止降级 `vibe`

**0.5 双维度交叉档位**

|  | 复杂度低 | 复杂度中 | 复杂度高 |
|---|---|---|---|
| ≤ 0.5 PD | vibe | vibe | plan（升档） |
| 1~3 PD | plan | plan | full（升档） |
| > 3 PD | full | full | full |

**0.6 输出分流卡片，等待用户确认（不得自动推进）**

```text
╔═══════════════════════════════════════════════════════╗
║  SDD 需求分流                                        ║
╠═══════════════════════════════════════════════════════╣
║  需求摘要：<一句话>                                  ║
║                                                     ║
║  预估 PD：<X> PD                                     ║
║  复杂度评分：                                        ║
║    ① 影响面       <0/1/2>  <理由>                   ║
║    ② 数据契约     <0/1/2>  <理由>                   ║
║    ③ 依赖耦合     <0/1/2>  <理由>                   ║
║    ④ 可逆性       <0/1/2>  <理由>                   ║
║    ⑤ 不确定性     <0/1/2>  <理由>                   ║
║    合计：<N> 分（低/中/高）                          ║
║                                                     ║
║  → 建议模式：<vibe / plan / full>                   ║
║    依据：<PD x.x + 复杂度 X 分>                      ║
╚═══════════════════════════════════════════════════════╝

请选择：
  [Y] 同意建议模式
  [1] 改为 vibe —— 跳过 SDD，直接写代码
  [2] 改为 plan —— 轻量规格模式
  [3] 改为 full —— 完整四阶段
  [E] 我来调整 PD 或评分
```

**0.7 写入状态**

用户确认后，写入 `.agents/sdd-state.json` 的 `triage` 字段：

```json
{
  "triage": {
    "mode": "vibe | plan | full",
    "pd": 2.0,
    "complexityScore": 5,
    "complexityLevel": "low | medium | high",
    "dimensions": {
      "scope": 1,
      "contract": 2,
      "coupling": 1,
      "reversibility": 0,
      "uncertainty": 1
    },
    "recommendation": "plan",
    "userChoice": "plan",
    "overrideReason": "<若改了建议档位，必填>",
    "timestamp": "<ISO 时间戳>"
  }
}
```

**0.8 根据档位路由**

| 档位 | 后续动作 |
|------|---------|
| `vibe` | 提示「建议直接写代码，无需 SDD」，退出 start 流程 |
| `plan` | 阶段一保留（proposal）→ 阶段二走轻量规格分支（单文件 `spec.md`）→ 阶段三 build → 阶段四 close |
| `full` | 完整四阶段流程（proposal → spec/design/tasks → build → close） |
