# Build: 执行实现

## 目标

将 `plan-ready.md` 或轻量 `spec.md` 转化为真实实现。先生成原子化执行计划，再按任务轨道执行：`[code]`、`[integration]`、`[manual]`、组件前置任务。

## 前置条件

- `openspec/changes/<变更名>/plan-ready.md` 存在，或
- `openspec/changes/<变更名>/spec.md` 存在（plan 模式）

如果不满足，提示：
> 「还没生成可执行规格。请先完成 /sdd spec。」

## 流程

### 步骤零：依赖检测

执行前检查以下依赖是否可用：

| 依赖 | 检测方式 | 不可用时 |
|------|----------|----------|
| Superpowers writing-plans | 对应 skill 是否存在 | 降级为手动拆解规格步骤，逐条执行 |
| OpenSpec CLI | `openspec` 命令是否可执行 | 不影响 build 阶段，但 close 阶段归档可能需手动移动目录 |

如果 Superpowers 不可用，提示用户：
> 「Superpowers 未安装，build 将使用手动执行模式。安装后体验更佳。」

### 步骤一：检测状态与信息收集

检查以下文件确定当前状态：

| 检查 | 怎么查 | 结果 |
|------|--------|------|
| 有活跃变更？ | `openspec/changes/` 下非 archive 子目录 | 找到变更名 |
| 有 `plan-ready.md` 或 `spec.md`？ | 变更目录下是否存在 | 不存在 → 提示先 spec |
| 实现已开始？ | 变更目录下是否有 `impl-plan.md` | 已开始 → 断点恢复 |

如果有多个活跃变更，列出并让用户选择。

**状态确认后，立即收集执行所需信息：**

- 若执行计划中包含 `[integration]` 任务，记录其对应的外部平台、工单、环境或发布目标
- 同步运行基线检查，记录当前代码库状态

**基线检查：**

运行项目实际命令，如：

```bash
npm run typecheck && npm run lint
```

将输出记录到变更目录下的 `build-baseline.log`。若存在存量错误，在执行摘要中标注“存量问题，非本次引入”，不阻塞 build。

#### 步骤一·补强检查：一致性回溯校验（强制）

build 阶段开始前，对 spec 阶段产物做一次回溯校验，避免 spec 误判遗留到实现阶段：

1. 读取 `design.md` / `prd-checklist.md` / `tasks.md`
2. 对 `design.md` 中记录的受影响文件、模块、入口做一次重扫
3. 若本项目依赖 feature flag、外部配置、平台状态或灰度信息，查询其最新状态
4. 不一致时：

| 不一致类型 | 处理 |
|---|---|
| 发现 spec 阶段未识别的新入口 / 新模块 | 暂停 build，回到 `/sdd spec amend` |
| 外部配置状态发生变化，原任务边界可能失效 | 暂停 build，回到 `/sdd spec amend` |
| 外部配置从启用变成停用 | 可继续 build，但在执行摘要中标注风险 |
| 完全一致 | 继续步骤二 |

### 步骤二：断点恢复

如果检测到变更目录下已有 `impl-plan.md`，检查其中 checkbox 状态：

- 全部勾选 → 提示实现已完成，建议 `/sdd close`
- 部分勾选 → 从未完成 task 继续执行
- 外部任务已有记录号或回执 → 跳过重复下发，直接读取状态
- 无勾选 → 从头开始

### 步骤三：生成原子化执行计划

在生成计划前，调用 `Karpathy编码原则` 作为本次执行的行为约束。

使用 `plan-ready.md` 或轻量 `spec.md` 作为输入，生成 `impl-plan.md`。

输出格式建议：

```text
执行计划：

[integration] 任务 1：[任务描述]
  步骤 1.1：具体操作
  步骤 1.2：预期变更
  步骤 1.3：验证方式
  执行器：外部平台 / 集成适配器 / 人工

[code] 任务 2：[任务描述]
  步骤 2.1：具体操作
  步骤 2.2：预期变更
  步骤 2.3：验证方式（含测试方法）
  执行器：Code Agent
```

对描述模糊或影响面不清晰的任务，主动向用户提问，每次只问一个问题。

#### 依赖分析 Pass

扫描 `impl-plan.md` 中所有 `[integration]` 和 `[code]` 任务，识别外部任务对代码任务的参数依赖，产出前置 code task 集合。

在文件头部追加一个前置集合声明区块，例如：

```text
## Integration 下发前置条件

> 所有 [integration] 任务将在以下 [code] task 全部完成后统一下发：
> - ⚙️ code task-2（原因：定义 onSubmit action，外部 schema 依赖此字段）
> - ⚙️ code task-5（原因：新增 goodsList params，外部模块依赖此字段）
```

若无依赖：

```text
## Integration 下发前置条件

> 未检测到全码参数依赖，[integration] 任务将在组件前置链完成后立即统一处理。
```

输出执行计划摘要：

```text
执行计划摘要：
- 任务总数：[N] 个
- [comp-integration] 任务：[N] 个 → 外部组件/资产链路（串行前置）
- [comp-code] 任务：[N] 个 → 全码组件链（前置）
- [integration] 任务：[N] 个 → 外部平台或跨系统任务
- [code] 任务：[N] 个 → 代码实现任务
- [manual] 任务：[N] 个 → 人工处理
- 预估执行时长：约 [N] 分钟
```

### 步骤三点五：组件前置执行链（有 `[comp-code]` 或 `[comp-integration]` 时）

扫描 `impl-plan.md`，若存在组件级任务，必须在主工程任务执行前完成。

#### 全码组件链（[comp-code]）

1. 先做 TDD 改造，不立即发布
2. 同库全部任务完成后，再统一发布与升版
3. 若项目没有自动发布脚本，至少明确记录手动发布步骤

#### 外部组件链（[comp-integration]）

1. 完成组件改造与验证
2. 发布组件资产或同步配置
3. 若需要升级外部平台版本，统一批量处理
4. 全部失败时强制暂停；部分失败时由用户决定是否继续

### 步骤四：执行实现（多轨）

#### 阶段 A：外部任务统一处理（有 `[integration]` 任务时）

所有 `[integration]` 任务只处理一次，禁止重复派发。

执行时机：
- 有前置 code task → 等前置全部完成后处理
- 无前置 code task → 组件前置链完成后立即处理

处理方式：
1. 扫描所有 `[integration]` 任务，收集目标平台、环境、动作与依赖
2. 如果项目已配置连接器/脚本，则统一调用
3. 如果没有自动化能力，则生成统一的人工执行清单，并记录回执位
4. 输出每个外部任务的状态与回执

#### 阶段 B：全码任务逐一执行（[code] 任务）

每个 task 执行前，先过一遍 `Karpathy编码原则` 的快速自检清单：

```text
□ 我理解这个 task 要做什么了吗？假设是否已明确？
□ 我计划的实现方案是最简单的方案吗？
□ 我是否只改动了 task 要求的代码，没有额外触碰？
□ 我的成功标准是可验证的吗（测试/命令/截图）？
```

执行规范：

1. 含业务逻辑的 `[code]` 任务尽量执行红灯 → 绿灯 → 重构循环
2. 纯 UI / 纯配置类任务可以用 lint + 人工验证代替测试先行
3. 每个 `[code]` 任务完成后只执行 `git add -A`
4. 严禁在 build 阶段自行 `git commit` / `git push`
5. 编译 / 测试不通过不进入下一个 task

仅在以下情况主动暂停：
- 测试 / 编译失败
- 发现规格歧义，无法独立决策
- 涉及不可逆操作

暂停时输出：

```text
⚠️ 暂停原因：[原因描述]
任务进度：[已完成数] / [总任务数]
当前任务：[任务描述]

请决策：
  A. 提供补充信息后继续
  B. 跳过此任务（标记待处理）
  C. 回退到 spec 阶段
```

### 步骤五：执行完成

所有 task 完成后，先运行：

```bash
git diff --stat
```

将实际改动范围与 `impl-plan.md` 中的预期文件列表对比，再输出执行摘要：

```text
执行摘要：
- 完成任务：[N] 个
- [integration] 任务回执：[编号或状态]
- 通过测试：[N] 个
- 实际改动文件（git diff --stat）：[N] 个文件
  - 计划预期：[N] 个文件 ✅ / ⚠️ 超出预期 [+N] 个

⚠️ 当前处于无 commit 模式，暂存变更待人工 review 后提交
```

提示用户：
> 「所有实现任务已完成。接下来可以用 /sdd close 验证一致性并归档。」

## 异常处理

| 异常情况 | 处理方式 |
|----------|----------|
| 子 Agent 执行失败 | 暂停并告知用户，等待决策：重试 / 跳过 / 回退 spec |
| 测试无法通过 | 暂停，展示失败原因，等待用户介入 |
| 外部任务处理失败 | 展示错误原因，询问用户：重试 / 跳过 / 回退 spec |
| 依赖推断无法判断 | 保守纳入前置集合，并在摘要中标注 |
| 组件库发布失败 | 暂停，等待用户决策：重试 / 手动发布后继续 |

## 关键原则

- 不允许在 build 阶段偷偷改需求
- `plan-ready.md` / `spec.md` 是锁定的执行输入
- writing-plans 只做任务拆解与排序，不重新定义需求
- 断点恢复依赖文件系统状态，不依赖会话记忆
- 本阶段输出的执行计划是唯一权威输入；若发生人工干预，必须重新确认计划后再继续
