# Brainstorming: 深度设计

## 目标

通过多轮对话，深入探索需求、方案取舍和架构决策。产出比 proposal 更完整的需求描述和设计方向。

## 流程

### 1. 理解背景

在提问或分析之前，先按以下顺序获取上下文：

1. 读取当前变更与已有文档（`proposal.md`、相关 `design.md`、近期变更）
2. 读取规则层与项目上下文（`vertical-knowledge/rules/`）
3. 如有知识库，再查询相关模块知识
4. 对知识库无法回答的部分，再进行代码探索

> 不要一上来直接写方案，也不要先动代码。

### 2. 逐个提问

一次只问一个问题，逐步深入。问题类型可以包括：

- 目的：这个功能的核心用户场景是什么？
- 取舍：A 更简单但扩展性差，B 更灵活但复杂。你倾向哪个？
- 边界：如果 X 情况发生，期望行为是什么？
- 优先级：这几个需求里，哪个最重要？

### 3. 提出 2-3 种方案

基于讨论，提出 2-3 种实现方案，附上取舍分析。推荐一种并说明理由。

### 4. 确认设计

用户选定方案后，整理设计要点并与用户确认：

> 「确认的设计方向：[方案名]。核心决策：[2-3 条]。这样对吗？」

### 5. 创建或更新变更提案

用户确认后，若还没有变更：

```bash
openspec new <变更名>
```

如果 CLI 不可用：

```bash
mkdir -p openspec/changes/<变更名>/specs
```

将确认的需求描述和设计方向写入 `proposal.md`，必要时补写 `design.md` 初稿。

### 6. 提示下一步

> 「需求和设计方向已记录。接下来可以用 `/sdd spec` 生成完整规格。」

## 注意

- 不要写代码
- 不要跳过取舍讨论直接给答案
- 如果项目很大，建议先拆成独立子项目
- 允许用户改变方向，不要过早锁定
- 不要把一次 brainstorming 直接推进到 build
